# 07-01. SDN 기본 개념

> 네트워크 엔지니어 면접 시나리오

> CCNA 200-301 수준의 SDN(Software-Defined Networking) 관련 면접 질문과 답변

---

### 면접관: "전통적인 네트워크와 SDN의 가장 큰 차이점이 무엇인지 설명해 주시겠습니까?"

네, 전통적인 네트워크에서는 각 네트워크 장비(스위치, 라우터)가 **Control Plane과 Data Plane을 모두 자체적으로 보유**합니다. 즉, 각 장비가 독립적으로 경로를 계산하고, 트래픽을 전달하는 구조입니다.

반면 SDN에서는 **Control Plane을 중앙의 SDN Controller로 분리**합니다. 네트워크 장비는 Data Plane(트래픽 전달)만 담당하고, 경로 결정 등의 지능적인 작업은 컨트롤러가 일괄 처리합니다.

#### 핵심 비교

| 구분 | 전통적 네트워크 | SDN |
|------|----------------|-----|
| Control Plane | 각 장비에 분산 | 중앙 컨트롤러에 집중 |
| 설정 방식 | 장비별 CLI 접속 | 컨트롤러를 통한 중앙 관리 |
| 유연성 | 변경 시 장비별 작업 필요 | 프로그래밍으로 일괄 변경 가능 |
| 확장성 | 장비 수 증가 시 관리 복잡 | 컨트롤러가 통합 관리 |
| 자동화 | 제한적 (스크립트 수준) | API 기반 완전 자동화 가능 |

---

### 면접관: "네트워크의 3가지 Plane에 대해 각각 역할을 설명해 주세요."

네트워크 장비의 기능은 크게 세 가지 Plane으로 나뉩니다.

**1. Management Plane (관리 평면)**
- 네트워크 장비를 **설정하고 모니터링**하는 데 사용됩니다
- SSH, SNMP, Syslog, NTP 등이 해당됩니다
- 관리자가 장비에 접근하여 구성을 변경하는 영역입니다

**2. Control Plane (제어 평면)**
- **경로를 결정**하는 역할을 합니다
- OSPF, EIGRP, BGP 같은 라우팅 프로토콜이 여기서 동작합니다
- ARP, STP 등 네트워크 제어 프로토콜도 포함됩니다
- "어디로 보낼지"를 결정하는 두뇌 역할입니다

**3. Data Plane (데이터 평면 / Forwarding Plane)**
- 실제 **사용자 트래픽을 전달**하는 역할입니다
- Control Plane이 만든 라우팅 테이블, MAC 테이블을 기반으로 패킷을 포워딩합니다
- "결정된 경로로 실제로 보내는" 역할입니다

SDN에서는 이 중 **Control Plane을 중앙 컨트롤러로 분리**하는 것이 핵심 개념입니다.

---

### 면접관: "SDN Controller의 Northbound API와 Southbound API는 각각 무엇인가요?"

SDN Controller는 위아래로 두 가지 인터페이스를 가집니다.

**Southbound API (SBI)**
- SDN Controller와 **네트워크 장비(스위치, 라우터) 사이**의 인터페이스입니다
- 컨트롤러가 장비에 명령을 내리고, 장비 상태를 수집합니다
- 대표적인 프로토콜: **OpenFlow**, NETCONF, RESTCONF, OpFlex
- 쉽게 말해 컨트롤러가 "아래쪽 장비"와 통신하는 방법입니다

**Northbound API (NBI)**
- SDN Controller와 **애플리케이션 사이**의 인터페이스입니다
- 관리자나 자동화 프로그램이 컨트롤러에 명령을 전달합니다
- 주로 **REST API** 형태로 제공됩니다
- 쉽게 말해 "위쪽 애플리케이션"이 컨트롤러와 통신하는 방법입니다

```
┌─────────────────────────┐
│   애플리케이션 / 스크립트   │
└──────────┬──────────────┘
           │ Northbound API (REST API)
┌──────────▼──────────────┐
│     SDN Controller      │
└──────────┬──────────────┘
           │ Southbound API (OpenFlow, NETCONF)
┌──────────▼──────────────┐
│   네트워크 장비 (스위치)    │
└─────────────────────────┘
```

---

### 면접관: "OpenFlow에 대해 간단히 설명해 주세요. 실무에서 많이 사용되나요?"

OpenFlow는 **SDN에서 가장 초기에 등장한 Southbound API 프로토콜**입니다.

#### 핵심 개념

- SDN Controller가 스위치의 **Flow Table을 직접 프로그래밍**할 수 있게 해줍니다
- 스위치는 Flow Table에 따라 패킷을 전달, 드롭, 수정할 수 있습니다
- 매칭 조건(Match)과 동작(Action)으로 구성된 Flow Entry를 사용합니다

```
Flow Entry 예시:
Match: src_ip=10.0.1.0/24, dst_port=80
Action: forward to port 3

Match: src_ip=192.168.1.0/24
Action: drop
```

실무에서는 순수 OpenFlow보다는 벤더별 SDN 솔루션이 더 많이 사용됩니다. 예를 들어 Cisco는 **OpFlex** 프로토콜을 ACI에서 사용하고, **NETCONF/RESTCONF** 같은 표준 프로토콜도 널리 쓰입니다. 하지만 OpenFlow는 SDN의 기본 개념을 이해하는 데 매우 중요한 프로토콜입니다.

### 면접관: "SDN을 도입하면 어떤 장점이 있을까요? 구체적으로 3가지 이상 말씀해 주세요."

SDN 도입의 주요 장점은 다음과 같습니다.

**1. 중앙 집중식 관리 (Centralized Management)**
- 수백 대의 장비를 개별 접속하지 않고 하나의 컨트롤러에서 관리할 수 있습니다
- 네트워크 전체의 토폴로지를 한눈에 파악할 수 있습니다

**2. 프로그래밍 가능성 (Programmability)**
- API를 통해 네트워크를 코드로 제어할 수 있습니다
- 반복 작업을 자동화하여 운영 효율이 크게 향상됩니다

**3. 빠른 변경 및 배포 (Agility)**
- 새로운 VLAN 추가, 정책 변경 등을 몇 분 안에 전체 네트워크에 적용할 수 있습니다
- 전통적 방식에서는 장비별로 수동 작업이 필요했던 것을 일괄 처리합니다

**4. 일관성 있는 정책 적용 (Consistency)**
- 컨트롤러가 모든 장비에 동일한 정책을 적용하므로 설정 불일치(configuration drift) 문제가 줄어듭니다

**5. 비용 절감**
- 운영 자동화로 인건비 절감, 장애 시간 감소로 비용이 줄어듭니다

### 면접관: "Cisco ACI에 대해 아는 대로 설명해 주세요."

Cisco ACI(Application Centric Infrastructure)는 Cisco의 **데이터센터용 SDN 솔루션**입니다.

#### 핵심 개념

- **APIC(Application Policy Infrastructure Controller)**가 SDN Controller 역할을 합니다
- **Nexus 9000 시리즈** 스위치를 사용하여 Spine-Leaf 아키텍처를 구성합니다
- **OpFlex** 프로토콜을 Southbound API로 사용합니다 (OpenFlow가 아님)
- 정책 기반(Policy-Based) 네트워킹을 구현합니다

**ACI의 주요 구성 요소:**

| 구성 요소 | 역할 |
|----------|------|
| APIC | 중앙 컨트롤러, 정책 관리 |
| Spine Switch | 백본 역할, Leaf 스위치 간 연결 |
| Leaf Switch | 서버/외부 네트워크 연결 포트 제공 |
| Tenant | 논리적 격리 단위 (가상 네트워크) |
| EPG (Endpoint Group) | 동일 정책을 적용받는 엔드포인트 그룹 |
| Contract | EPG 간 통신 정책 정의 |

ACI에서는 "이 서버 그룹은 저 서버 그룹과 HTTP 통신을 허용한다"와 같은 **의도(Intent) 기반 정책**을 정의하면, APIC이 이를 각 스위치에 자동으로 배포합니다. CCNA 수준에서는 ACI의 기본 아키텍처와 APIC의 역할을 이해하는 것이 중요합니다.

### 면접관: "SDN과 전통적 네트워크 중 어떤 환경에서 SDN이 더 적합한지 예를 들어 설명해 주세요."

SDN이 특히 적합한 환경은 다음과 같습니다.

**대규모 데이터센터**
- 수백~수천 대의 서버와 스위치를 관리해야 할 때, 장비별 CLI 설정은 비현실적입니다
- Cisco ACI나 VMware NSX 같은 SDN 솔루션이 효과적입니다

**클라우드 환경**
- 고객 요청에 따라 네트워크를 빠르게 프로비저닝해야 하는 클라우드 서비스 제공자에게 필수적입니다
- AWS, Azure 등의 가상 네트워크가 SDN 기반으로 동작합니다

**빈번한 변경이 필요한 환경**
- 개발/테스트 환경에서 네트워크 구성이 자주 바뀌는 경우
- CI/CD 파이프라인과 연동하여 자동으로 네트워크를 구성할 수 있습니다

반면, 소규모 네트워크(스위치 5~10대 수준)에서는 전통적 방식이 더 단순하고 비용 효율적일 수 있습니다. SDN 컨트롤러 도입 비용과 학습 비용 대비 얻는 이점이 크지 않을 수 있기 때문입니다.

핵심은 **네트워크 규모와 변경 빈도**를 고려하여 적절한 방식을 선택하는 것입니다.

---

### 면접관: "마지막으로, SDN에서 컨트롤러가 장애를 일으키면 네트워크에 어떤 영향이 있을까요?"

이것은 SDN의 중요한 고려사항입니다.

**컨트롤러 장애 시 동작:**
- 대부분의 SDN 솔루션에서 컨트롤러가 다운되더라도 **기존에 설치된 Flow Table은 유지**됩니다
- 따라서 이미 학습된 트래픽은 계속 전달됩니다
- 하지만 **새로운 Flow 추가나 정책 변경은 불가능**해집니다

**대비 방안:**
- SDN 컨트롤러를 **클러스터로 구성**하여 고가용성(HA)을 확보합니다
- 예를 들어 Cisco APIC은 보통 **3대의 컨트롤러를 클러스터**로 운영합니다
- 1대가 장애를 일으켜도 나머지 2대가 서비스를 지속합니다

이는 전통적 네트워크의 분산 아키텍처 대비 SDN의 약점이 될 수 있지만, 적절한 이중화 설계로 충분히 해결 가능한 부분입니다.
